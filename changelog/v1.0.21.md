# RxFHIR v1.0.21

## Added

### App Update Checker
- The About page now includes an **App Updates** section.
- A **Check for Updates** button queries the GitHub Releases API (`/repos/swiftruru/rx-fhir/releases/latest`) and compares the latest tag against the running version using semver.
- Three result states are displayed inline:
  - **Up to date** — green checkmark confirmation.
  - **Update available** — amber banner showing the new version number and a **View on GitHub Releases** button that opens the Releases page in the system browser.
  - **Check failed** — red error message (network timeout, API error, etc.).
- On startup, a background check runs automatically after a 5-second delay. If a newer version is found, a small amber dot appears next to the version number in the Hero block as a non-intrusive indicator.
- The startup check only notifies the renderer when `status === 'update-available'`; "up to date" and error results are silently discarded to avoid disrupting the user.

### Architecture
- New `src/main/updater/` module (three files):
  - `updateChecker.ts` — GitHub API fetch using Node.js built-in `https`, 8-second timeout, semver comparison, skips drafts and pre-releases.
  - `updateIpc.ts` — registers the `app-update:check` IPC handler and `scheduleStartupUpdateCheck()`.
  - `index.ts` — unified re-export.
- `preload.ts`: exposes `checkForUpdates()` (invoke) and `onUpdateResult()` (listener with automatic cleanup) on `window.rxfhir`.
- `src/renderer/types/electron.d.ts`: adds `UpdateCheckResult` and `UpdateStatus` types and bridge method signatures.
- i18n: adds `about.update.*` keys in both `en` and `zh-TW` settings namespaces.

### Design decisions
- No `electron-updater` dependency: macOS has no code signing/notarization, Windows Portable cannot be auto-updated, and CI uses `--publish never` (no `latest.yml` metadata file).
- No GitHub token required: the public releases API works without authentication.
- No CI or `electron-builder.yml` changes needed.
- All platforms (macOS, Windows installer, Windows portable, Linux) use the same detection-and-redirect strategy.
