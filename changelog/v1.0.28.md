# RxFHIR v1.0.28

## Fixed

### Feature Showcase no longer triggers Live Demo FHIR submissions

When Feature Showcase started while Live Demo was running or paused, the two
systems could collide. The showcase navigates the Creator to step 10
(Composition/Extension), which renders the form and registers its `submit`
controller in `liveDemoStore`. The Live Demo async loop — still running in the
background — would detect the newly registered controller and immediately call
`onSubmit()`, firing a real `POST Composition` to the FHIR server with mock
patient IDs (`showcase-primary-patient`) that do not exist on the public HAPI
server. The result was one or more 400 / 404 error toasts appearing on screen
during the showcase tour.

- `FeatureShowcaseRunner.tsx`: reads `useLiveDemoStore.getState().status` on
  every showcase start; if Live Demo is not `idle`, calls `stop()` before
  proceeding — this terminates the async loop and unregisters all form
  controllers, preventing any background submission

### FHIR 410 Gone recovery in findOrCreateDetailed

The public HAPI FHIR server can leave stale entries in its identifier search
index after soft-deleting a resource. The existing `findOrCreateDetailed`
pattern was:

1. `GET /{type}?identifier=...` → 200, 0 entries (deleted resource hidden from
   search)
2. `POST /{type}` → 412 (HAPI still considers the deleted resource a duplicate
   and returns its ID in the error)
3. `GET /{type}/{id}` → **410 Gone** → error thrown to the user

The 410 error surfaced as "Resource was deleted at ..." in the Creator's
Extension (Basic) step and could affect any resource type that uses the
`findOrCreateDetailed` reuse path.

- `fhirClient.ts` — `findOrCreateDetailed`: the `fetchResourceById` call inside
  the duplicate-recovery branch is now wrapped in a try/catch; when a 410 error
  is detected (error message contains `(410)`), the function issues a
  `PUT /{type}/{id}` instead — per FHIR spec §3.1.0.6, PUT on a deleted
  resource creates a new current version, effectively resurrecting it; this
  allows the submission to succeed without surfacing the stale-index error to
  the user

## Technical changes

- `src/renderer/components/FeatureShowcaseRunner.tsx`: imports
  `useLiveDemoStore`; adds Live Demo stop guard at the top of the `runId`
  effect before backup and snapshot injection
- `src/renderer/services/fhirClient.ts`: `findOrCreateDetailed` — wraps
  `fetchResourceById` in the duplicate-ID recovery path with a try/catch; on
  410 falls through to `putResource` with the original body
