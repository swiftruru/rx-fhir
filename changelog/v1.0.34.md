# RxFHIR v1.0.34

## Changed

### Renderer and Electron ownership boundaries are now explicit

This release reorganizes the app into clearer ownership layers so future
changes no longer need to guess whether code belongs to app shell, feature,
shared UI, or FHIR domain logic.

- renderer code is now structured around `app`, `features`, `domain`, and
  `shared`
- Electron main-process responsibilities are split into focused IPC and
  desktop service modules under `src/main/ipc` and `src/main/services`
- stable cross-process and shared domain contracts now live under
  `src/shared/contracts`
- the README and architecture docs now describe the current repository shape
  instead of the pre-refactor structure

## Added

### Regression guardrails for refactor-safe releases

This release adds a basic automated safety net so architecture changes can be
verified before tagging new versions.

- `vitest` + `jsdom` are now included for unit-level regression coverage
- new tests cover bundle building, bundle file parsing, search result shaping,
  and Creator draft persistence / hydration
- UX manual testing docs now include a maintained checklist and a smoke report
  under `docs/ux/manual-testing/`

## Fixed

### Creator restored-draft banner now appears after hydration

Persisted Creator drafts were being restored correctly, but the one-time
"restored draft" banner could fail to appear after app reload because the
hydration-derived UI flags were not being computed robustly during automatic
store hydration.

- Creator draft hydration state is now derived during persist merge
- the restored banner now appears together with the recovered draft values
- regression coverage was added for both explicit rehydrate and initial module
  hydration paths
