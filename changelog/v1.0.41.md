# RxFHIR v1.0.41

## Fixed

### Document Bundle submission was silently blocked by fullUrl/reference mismatch

After pressing "組裝並提交 Document Bundle", the server-side `$validate`
returned many ERROR-level issues because `urn:uuid:` fullUrls could not be
matched to `ResourceType/id` references under FHIR bundle resolution rules.
The audit report set `hasBlockingErrors: true` and silently prevented the
actual Bundle POST — pressing the button had no visible effect.

- bundle entries for server-created resources now use absolute server URLs
  (`{baseUrl}/{ResourceType}/{id}`) as their `fullUrl`, so
  `ResourceType/id` references resolve correctly under FHIR bundle rules
- the Composition entry (not yet on the server) continues to use a
  `urn:uuid:` identifier
- `serverBaseUrl` is now passed through `assembleDocumentBundle` to both
  the preview bundle and the submission bundle
- fullUrl-mismatch validator messages (which can still appear for the
  Composition's `urn:uuid:` entry) are now downgraded to info severity so
  they do not block submission

### Previous v1.0.39–v1.0.40 fixes also included in this release

- HAPI-2840 duplicate Composition error: the Composition is no longer
  POSTed individually before the Document Bundle
- HAPI-0505 invalid reference format: references keep `ResourceType/id`
  format instead of being rewritten to `urn:uuid:`
- Error toast duration extended from 4.2 s to 10 s
